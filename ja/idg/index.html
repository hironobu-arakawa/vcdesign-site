IDG — 日本語入口

曖昧な入力は、ここで止める。

IDG（Interface Determinability Gate）は、 「説明できないもの」「責任が曖昧なもの」「境界を越えるもの」を システムに入れないための 最小のインターフェース・ゲート です。

これはガバナンスではありません。 判断もしません。 決定不能なものを通さないだけです。

IDGとは（短く）

IDGは、外部から入ってくる入力が 説明できるか（Explainability） 追跡できるか（Auditability） 責任が明確か（Ownership） 境界内か（Scope） を確認し、曖昧なら「通さない」ゲートです。

これは「AIを安全にする仕組み」ではなく、 曖昧性が境界を越えて責任が消えるのを防ぐための最小構造です。

なぜIDGが必要なのか

  • 1. 曖昧な入力が「勝手に意味を持つ」から 未定義のフィールドや曖昧な文脈が、後工程で“事実扱い”される。
  • 2. AIが「決めていないこと」を勝手に補完するから 推測や補完が責任境界を越えると、説明不能な判断が生まれる。
  • 3. 責任の所在が曖昧なまま進むと、後戻りできないから 境界で止めないと、曖昧さがそのまま実行に入ってしまう。

IDGがブロックする条件（4つの曖昧性）

  • Auditability（追跡不能） 後から「なぜこの入力が来たのか」が説明できない。
  • Explainability（説明不能） 拒否理由を第三者に構造として説明できない。
  • Ownership（責任不明） 誰がこの入力の意味を引き受けるのか不明。
  • Scope（境界外） 宣言された境界を越えている。

IDGは「Fail-Closed」

IDGは「曖昧なら通さない」が原則です。 推測しません。補完しません。 曖昧なまま進めることが最大のリスクだからです。

  • デフォルトは拒否 曖昧なら通さない。
  • 理由を返す 拒否理由は構造として返す（感情や物語ではなく）。
  • 補完しない AIが勝手に「良かれと思って」補完しない。

VCDesignの中でのIDGの位置

IDGは「最も外側の境界」を守るゲートです。 RPやBOAより前にあります。

  • IDG → 外部入力の曖昧性を止める （External → System Boundary）
  • RP → Meaning → Responsibility の昇格を守る （判断前インターロック）
  • BOA → Fact / Meaning / Responsibility を分離する （構築方法）
  • VCD → なぜ境界が必要なのかを定義する （哲学的コア）
