Interface Determinability Gate (IDG)

IDG (Interface Determinability Gate) is a lightweight interlock. It blocks undeterminable interactions at system boundaries before they cause responsibility leakage or unsafe coupling.

IDG is not governance.
IDG does not decide, interpret, or authorize. It only enforces minimum determinability at the boundary.

What IDG is

IDG is a fast, local gate for boundary inputs. It is evaluated at the interface where a system receives signals, data, or commands from outside its trusted domain.

  • checks determinability (can this be explained and audited?)
  • checks boundary integrity (does this violate declared scope?)
  • checks responsibility leakage (is ownership implicit or missing?)

If IDG conditions are not satisfied, the system must fail closed.

What IDG is not

IDG is not:

  • a decision mechanism
  • a policy engine
  • a replacement for human judgment
  • a way to “make AI safe” by post-hoc explanation
  • a substitute for RP (Resolution Protocol)

IDG defines hard stops and reject reasons, not outcomes.

Where IDG applies in VCDesign / BOA

BOA separates artifacts into:

  • Fact (immutable observations)
  • Meaning (interpretation / hypothesis / suggestion)
  • Responsibility (actions and commitments)

RP governs the promotion boundary:

Meaning → Responsibility (promotion)

IDG applies earlier, at the boundary of trust:

External / Untrusted Input → System Boundary

IDG can be deployed even when the larger governance model is not shared, because it does not claim authority — it only blocks what is undeterminable.

Fail-closed by design

IDG is fail-closed by default:

  • Default outcome: block
  • If determinability is insufficient: reject with a reason
  • If a boundary is violated: reject with an explicit boundary reference

This prevents silent coupling and accidental responsibility.

Design principles

  • Fast: minimal checks, low operational burden
  • Local: scoped to the interface, not system-wide governance
  • Explainable: every block has a clear, documentable reason
  • Non-authoritative: it does not “decide”; it only denies passage
  • Composable: multiple IDGs can exist at different boundaries

Relationship to RP

IDG and RP are complementary, not hierarchical:

  • IDG blocks early (boundary integrity and determinability)
  • RP governs promotion (Meaning → Responsibility)
  • IDG can exist without RP
  • RP assumes an explicit Responsible Domain; IDG does not

Specification

IDG is intended to be implementable as a simple gate (code, config, or contract clause) at any interface where determinability matters.

There is intentionally no single reference implementation. IDG is designed to be adapted locally, without assuming shared governance or a central authority.

  • Related: Resolution Protocol (RP)

If you implement VCDesign, treat IDG as a deployable interlock that keeps boundaries safe even when governance is incomplete.
